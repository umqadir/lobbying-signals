# Topic Extraction Problem Statement

## Goal

We're building a system to detect **newsworthy signals** in US federal lobbying disclosure data. The end product is alerts like:

> "Tariff lobbying hit an all-time high in Q1 2025, with 47 companies spending $12M—triple the 5-year average"

To generate these signals, we need to know **what specific topics** each lobbying activity is about.

## The Data

We have ~200,000 lobbying activity records per year from the Senate's SOPR database. Each record has:

- **`description`**: Free-text field (50-500 words) describing what was lobbied on
- **`issue_code`**: One of 80 broad categories (e.g., "TRD" for Trade, "HCR" for Healthcare)

## The Problem

The existing `issue_code` is too coarse. "TRD" (Trade) contains everything from tariffs to export controls to trade agreements to sanctions. We can't detect that *tariff* lobbying specifically is spiking.

We need to extract **1-3 specific, searchable topics** from each description. Good topics are:
- Specific enough to be meaningful ("tariffs" not "trade")
- General enough to aggregate across filings ("tariffs" not "steel tariffs Section 232")
- Consistent terminology (don't have both "drug pricing" and "pharmaceutical costs")

### Example Outputs

| Description (truncated) | Extracted Topics |
|------------------------|------------------|
| "tariffs on steel and aluminum imports; Section 232..." | `["tariffs", "section 232", "steel imports"]` |
| "Lower Drug Costs Now Act; Medicare Part D rebate..." | `["drug pricing", "medicare", "fda approval"]` |
| "AI safety research funding; NIST AI Risk Management..." | `["ai regulation", "section 230"]` |

## Constraints

| Constraint | Value |
|------------|-------|
| Records to process | ~200,000/year (need to backfill 5 years = 1M total) |
| Budget | Prefer <$50 total, ideally <$10 |
| Time | Hours acceptable, days is too slow |
| Quality | Topics must be specific enough for trend detection |
| Consistency | Same concept should map to same topic string |
| Updates | Need to process ~20K new records quarterly |

## What We've Tried

- **Zero-shot LLM prompting** (Gemini 2.0 Flash): Works well, ~$6.40 per 200K records, ~6 hours
- **Issue codes alone**: Too coarse, not useful for signal detection

## Open Questions

1. **Is an LLM the right tool?** Would a fine-tuned classifier, embedding clustering, or traditional NLP approach work better given our constraints?

2. **If LLM, which one?** We benchmarked Gemini. Options include Groq, DeepSeek, Cerebras, local models (Ollama/MLX on M4 Mac).

3. **Consistency problem**: How do we ensure "drug pricing" and "pharmaceutical pricing" don't both appear? Post-processing normalization? Constrained generation? Embedding-based deduplication?

4. **Taxonomy vs. open extraction**: Should we define a fixed taxonomy of ~200 topics and classify into it? Or extract freely and normalize after?

5. **Hybrid approaches**: Could we use embeddings for cheap clustering, then LLM only for ambiguous cases?

## Success Criteria

1. Topics are specific enough that we can detect "tariff lobbying is up 300%" (not just "trade lobbying is up 50%")
2. Topics are consistent enough that the same concept always gets the same label
3. Cost and time are reasonable for quarterly updates
4. Quality is good enough that a journalist would trust the categorization
